Chapter 06 ended on a confession. We had built an executor that could run *any* plan — scan or index at the leaves, nested-loop or hash or sort-merge at the joins, group and sort on top — every operator composing through one tiny `open`/`next`/`close` interface. We could hand it a tree and it would faithfully produce the answer. And that was exactly the problem: it would faithfully produce the answer for *any* tree we handed it, including a catastrophically bad one. We picked every operator ourselves, by hand, and every time we did it we deferred the real question with the same promise — *something else decides this.* Now collect the debt.

So take an ordinary query, the kind that turns up a thousand times a day:

```sql
SELECT c.name, o.total
FROM orders o JOIN customers c ON o.cust_id = c.id
WHERE o.total > 500 AND c.country = 'JP'
```

One query. Now count the plans. The `orders` filter can run as a sequential scan or, if there's an index on `total`, an index scan. The `customers` filter, likewise, scan or index on `country`. The join can be nested-loop, hash, or sort-merge. And the two tables can be joined in either order — `orders` on the outside or `customers` on the outside. Multiply it out and this trivial two-table query already has *dozens* of distinct physical plans. Add a third table and the join orderings alone explode combinatorially; a six-way join has hundreds of valid shapes. Every one of those plans returns the **identical answer** — they are all correct. And their costs differ not by ten percent but by **orders of magnitude**: the difference between a query that returns in a millisecond and the very same query, same rows, grinding for an hour.

Nobody writes the plan by hand. The part of the engine that does — the **optimizer** — takes the declarative SQL, considers the space of equivalent plans, and picks one. But look at the brutal constraint it works under: it has to choose *fast*, in a millisecond or two, and it has to choose *before the query runs*. It cannot try all the plans and keep the fastest — there are far too many, and running even one of them is the whole expense we're trying to avoid. So it must judge plans it will never execute. It has to look at a tree and *predict* what that tree would cost — guess how many rows the `total > 500` filter will pass, guess how big the join's output will be, guess whether the index is worth its random fetches — and stake the entire query's performance on those guesses.

That is the shape of the whole chapter, and it's worth committing to up front. We will build a tiny cost-based optimizer by hand, the same way we built the executor: in Python, in your browser, one piece at a time. A **cost model** that scores a plan from estimated page reads and CPU work (§2). A **cardinality estimator** that predicts how many rows each operator emits — the hard, central problem, the one every real optimizer lives or dies on (§3). The **access-path decision**, index versus scan, which is finally where Chapter 04's `EXPLAIN QUERY PLAN` gets explained (§4). The **join-order search** at the combinatorial heart (§5). And then an honest look at what happens when the estimates lie (§6) — because they do, and when they do, the plan is wrong.

The reveal that organizes everything: the optimizer is a search over plans guided by *predicted* cost, and prediction is the whole game. Build the predictor well and the engine flies. Build it badly and the executor you wrote in Chapter 06 will faithfully, obediently, run something a thousand times too slow.

</Section>

Start where the optimizer starts: with the fact that one query is *many* programs. We spent all of Chapter 06 insisting that a query compiles to a tree of operators. The thing we glossed is that it compiles to a tree *non-uniquely* — there is a whole **space** of trees, every one of them a faithful translation of the same SQL, every one producing the same rows.

Take the join query from §0 and write out just three of its plans. **Plan A**: sequentially scan `orders`, filter for `total > 500`, sequentially scan `customers`, filter for `country = 'JP'`, then hash-join the two filtered streams. **Plan B**: use an index on `customers.country` to fetch only the Japanese customers, then for each one do an index-nested-loop into an index on `orders.cust_id` to find their big orders. **Plan C**: scan both tables, sort each on the join key, sort-merge them. Three trees. Three completely different shapes — different leaf operators, different join algorithms, different orders. And here is the property that makes the optimizer's job *possible*: all three are **correct**. They return byte-for-byte the same result set. Correctness is shared across the entire plan space — it's invariant, a thing you get for free no matter which plan you pick.

What is *not* shared is cost. If most customers are Japanese and most orders are large, Plan A's scans-and-hash might be the cheapest. If only a handful of customers are in Japan and each has few orders, Plan B's index-nested-loop touches almost nothing and wins by a factor of thirty. Plan C's double sort might be pure waste — unless the output needs to be sorted anyway, in which case it's suddenly competitive. The point is the *gap*: across plans that all agree on the answer, the cost can swing by orders of magnitude. So the optimizer's whole job reduces to a single sentence: **choose well within a set of plans that already agree on the answer.** It never has to worry about getting the right rows — every candidate does that. It only has to worry about getting them cheaply.

To choose the cheap plan you need a number for "cheap," and you need it for a plan you are *not going to run*. So the cost model is a function that reads an operator tree and returns an estimated cost — a prediction of how much work executing that tree would take — computed purely from the tree's shape and some statistics about the tables. It never touches the data. It does arithmetic.

The arithmetic is built on the two costs we've spent two chapters learning to respect. The first is **page reads** — disk I/O, the dominant expense, the thing Chapter 04 taught us to count. A sequential scan reads *every* page of a table; an index scan reads a few pages to walk the B-tree plus a few more to fetch the matching rows. The second is **CPU work** — the per-row cost of evaluating a predicate, hashing a key, comparing for a sort, copying a column. Per row it's tiny, far cheaper than a page read, but multiplied across millions of rows it stops being negligible. So a serviceable cost model is just the sum of the two, rolled up over the whole tree:

> **cost ≈ (pages read) + cpu_per_row × (rows processed)**, summed over every operator in the plan.

Each operator contributes its own page reads and its own per-row CPU, and the costs roll up from the leaves to the root the same way the rows do. The `cpu_per_row` constant is small — something like 0.01 in "page-read units" — encoding the fact that processing a row in memory is far cheaper than fetching a page from disk. That single number is the model's exchange rate between CPU and I/O, and it's roughly the kind of constant Selinger's original System R optimizer tuned by hand.<Footnote n={1} />

Now watch what the formula does to the §1 plans. A seq scan of a ten-thousand-page table reads all ten thousand pages and grinds through, say, a million rows: cost ≈ 10,000 + 0.01 × 1,000,000 = 20,000. An index scan for a selective predicate reads maybe fifty pages and processes five hundred rows: cost ≈ 50 + 0.01 × 500 = 55. The model prefers the index scan by a factor of several hundred — and exactly mirrors what you watched happen in Chapter 06 when you swapped a `SeqScan` leaf for an `IndexScan` and the rows-scanned counter collapsed. The cost model is that collapse, written as a number you can compute *before* running anything.

One honest complication, because we earned it in Chapter 05. The model counts **logical** I/O — pages the plan *asks for*. But the buffer pool sits underneath, and a page that's already resident is a cheap memory hit, not a disk read. So the true cost depends on what's hot, which the optimizer can't fully know at planning time. Real optimizers paper over this with an estimate (Postgres has separate `seq_page_cost` and `random_page_cost` knobs, and a guess at cache hit rates), but the gap is real: the cost model estimates logical I/O and the buffer pool quietly muddies it. We'll keep counting logical pages and stay honest that it's an approximation.

Here is the hero and the heartbreak. Every cost in §2 depends on a row count, and every one of those row counts has to be *estimated* — predicted before the query runs, from statistics, about data the optimizer has not looked at. This is **cardinality estimation**, and it is, by a wide margin, the hardest and most consequential thing the optimizer does. Cost models are roughly mechanical. Estimation is where real optimizers succeed or fail.

Start with one predicate. The key quantity is **selectivity**: the fraction of rows a predicate passes. `WHERE active = true` might pass 0.9 of the rows; `WHERE age > 30` maybe 0.4; `WHERE id = 12345` essentially 1/N, almost nothing. Multiply selectivity by the table's row count and you have the estimate the cost model wanted. But where does the selectivity number itself come from? The optimizer can't run the predicate — that's the whole point. So it keeps **statistics**: gathered ahead of time (Postgres's `ANALYZE` does this), they record each table's row count, each column's number of distinct values, the min and max, and — the workhorse — a **histogram** of how the column's values are distributed.

A histogram chops a column's range into buckets and records how many rows fall in each. Picture the `age` histogram you'll meet in Figure 7.3: ten thousand users sorted into eight equi-width bands ten years wide, the counts piling up in the middle of the range — [20–30) and [30–40) are fattest, with a couple thousand rows each — and thinning toward the young and old ends. To estimate `age > 40`, you just add the buckets above forty: 1,743 + 1,082 + 770 + 196 = 3,791 rows out of about 10,000, a selectivity of roughly **0.38**. No data read — the answer came straight off the summary, and it's exact at a bucket boundary like forty (the approximation error only creeps in *inside* a bucket, where the histogram has to assume values spread evenly). (Equi-width buckets are the simple version; production systems use **equi-depth** buckets, each holding roughly the same *number* of rows so that dense regions get finer resolution. Either way the idea is the same: summarize the distribution once, read estimates off it forever.<Footnote n={3} />)

So far so good — a single predicate's selectivity is a reasonable estimate. The trouble starts when predicates and tables combine, because the optimizer has to *guess how they interact*, and its standard guess is the **independence assumption**: it assumes two conditions are unrelated, so their combined selectivity is the product of the two. `country = 'JP' AND language = 'ja'` gets estimated as sel(country) × sel(language) — which is badly wrong, because those columns are *correlated*: nearly everyone in Japan speaks Japanese, so the true combined selectivity is far higher than the product. Independence is the assumption that quietly breaks on real data.

And joins are where the breakage becomes catastrophic, because **errors compound multiplicatively through a chain of joins.** Estimating a join's output cardinality means estimating how many pairs of rows match, which again leans on independence and on the distinct-value counts — and the estimate for join number two takes the (already wrong) estimate from join number one as its *input*. So the errors don't add; they multiply. A modest 2× error per join — easy to incur — becomes 2 × 2 × 2 × 2 = **16×** off after a four-join chain. The estimate and the reality start together at the leaves and walk exponentially apart as you climb. And since the cost model (§2) feeds entirely on these counts, an estimate that's 16× wrong can hand the join-order search (§5) a cost that's 16× wrong, and the optimizer will confidently choose a plan built on a fantasy.

Then chain joins and watch the two numbers, estimate and actual, march apart: a small gap at one join, a yawning one at four, the error factor ticking up multiplicatively as the assumptions stack — a 2×-per-join error blowing out to 16× across four joins. That growing gap is the single most important picture in the chapter, and it's the chapter's whole danger in one line: **the plan is chosen from the estimate, but it pays the actual bill.** It is, quite literally, the reason production databases pour so much engineering into statistics.

Now we cash the promissory note from Chapter 04. Back then you ran `EXPLAIN QUERY PLAN` and watched SQLite print `SCAN users` for one query and `SEARCH users USING INDEX` for another — the same table, a different access path — and we said, flatly, that *who chooses* between them, and *why*, was the optimizer's job, deferred to a later chapter. This is the later chapter. The choice between a sequential scan and an index scan is a cost comparison, and now you can do the arithmetic yourself.

Set it up. For a predicate like `WHERE col = ?` or `col > ?`, the planner has (at least) two ways to find the matching rows. The **sequential scan** reads every page of the table and filters — its cost is fixed, roughly the table's page count, *regardless of how many rows match*. Read all ten thousand pages whether one row qualifies or nine thousand do. The **index scan** walks the B-tree to the matching keys and then fetches the corresponding rows — and its cost *scales with the number of matches*, because each match is (in the worst case) a random page fetch into the heap, and random fetches are the expensive kind. Few matches, few fetches, cheap. Many matches, many fetches — and a pile of random I/O can cost more than one orderly sequential sweep.

So the two cost curves cross. Plot cost against selectivity and the seq scan is a flat horizontal line — fixed at "read the whole table." The index scan starts near zero (a selective predicate fetches almost nothing) and rises with selectivity as the matches and their random fetches accumulate. They intersect at the **selectivity crossover**: below it the index wins, above it the seq scan wins. With a ten-thousand-page table, a selectivity of 0.001 makes the index cost ~30 against the scan's 10,000 — the index in a landslide. A selectivity of 0.5 makes the index cost something like 25,000 in scattered fetches against the scan's flat 10,000 — and now the "dumb" full scan is *the smart choice*. Same table, same index, opposite verdicts, and the only thing that changed was the estimated selectivity from §3. That is the whole story behind Chapter 04's `SCAN`-versus-`SEARCH` line: not a rule, not a syntax lookup, but the optimizer comparing two cost curves and reading off which is lower at the selectivity it estimated.<Footnote n={6} />

There's a wrinkle that shifts the crossover, and it's worth naming because Chapter 04 set it up too: the **covering index**. The index scan's expense is the *heap fetch* — after finding a key in the index, hop to the table to get the other columns. But if the index already contains every column the query needs (a covering index, Chapter 04 §6), there's no heap hop at all; you answer the query from the index alone. That knocks the random-fetch term out of the index's cost, so the index stays cheap to much higher selectivities and the crossover slides way over. This is why the same predicate can favor an index for one query (it's covered) and a seq scan for another (it isn't) — the columns the query asks for change the arithmetic.

  title="Join ordering"
  quiz={{

      { text: "The second order is cheaper only because C is listed last in the query; join order follows the FROM clause, so rewriting the SQL is what makes it tractable — no search is needed.", },
      { text: "Both orders cost the same total work since they produce the same final rows; the second just looks cheaper because intermediate size doesn't affect cost, only the final result size does.", },
    ],

>

Access-path selection chooses *how* to read one table. Join ordering chooses *in what sequence* to combine many — and it is the combinatorial heart of the optimizer, the part that makes the search genuinely hard. Because the number of orders explodes: joining N tables, there are N! ways to sequence them (more, once you allow bushy trees), and order matters enormously for one specific reason — **the size of each intermediate result feeds the cost of the next join.**

Make that concrete. Join A (10,000 rows), B (10,000 rows), and C (10 rows), where C joins very selectively to either of the others but A and B join to each other promiscuously. Consider the order (A⋈B)⋈C: the first join produces a *million-row* intermediate (A and B match almost everything), and then you join that monster against C. Now consider (A⋈C)⋈B: the first join is A against the tiny, selective C, producing maybe *ten* rows, and then you join those ten rows against B. Same three tables, same final answer — but the first order built a million-row intermediate and the second built a ten-row one, and since the cost of a join is driven by the size of its inputs, the second order is *thousands of times cheaper*. The whole art of join ordering is this: **join the selective things early, keep the intermediates small,** and never build a giant intermediate you're only going to filter down later. (And notice this all rests on §3 — "C is selective" is a cardinality estimate. Get that estimate wrong and you'll cheerfully pick the million-row order.)

So the optimizer must search the orderings for a cheap one — but N! is hopeless to enumerate directly; a ten-way join has over three million orderings, and that's before counting algorithm and access-path choices at each step. The breakthrough is **System R's**, from Selinger's 1979 paper, and it's two ideas.<Footnote n={1} /> First, restrict the search to **left-deep** plans — trees where each join's right input is always a base table, so you're always joining an accumulated intermediate against one more table. That throws away the bushy shapes and shrinks the space enormously while keeping the orders that matter in practice. Second, search what's left with **dynamic programming.** The insight: the cheapest way to join a *set* of tables doesn't depend on how you got there, only on which tables are in the set. So build up by subset size — find the best plan for every pair of tables, then use those to find the best plan for every triple, then every quadruple — reusing the smaller solutions instead of recomputing them. DP turns the factorial search into something exponential-but-manageable (2^N subsets), and for the table counts most queries have, it's fast.

When N gets genuinely large — a twenty-way join, where even 2^N is too much — the DP itself becomes too expensive, and optimizers fall back to **heuristics**: greedy algorithms that just keep joining the cheapest-looking next table, or randomized/genetic search (Postgres switches to its "genetic query optimizer" past a threshold). These don't guarantee the optimal order; they settle for a good-enough one found in reasonable time. The modern extensible optimizers — the **Cascades** framework and its descendants — generalize the whole search into a rule-driven exploration of the plan space, but the core bet is unchanged: explore enough of the space to find a cheap plan, cleverly enough to avoid exploring all of it.<Footnote n={4} />

We've built the whole machine — cost model, estimator, access-path chooser, join-order search — so now be honest about how it breaks, because in production it breaks all the time, and always in the same place. The cost model is roughly fine. The search is roughly fine. **The estimates are where it dies**, and §3 already told you why: cardinality errors compound, and they feed everything downstream.

Trace one failure end to end. Suppose a query filters on two correlated columns — `country = 'JP' AND language = 'ja'` — and the optimizer, assuming independence, multiplies the two selectivities and estimates the join's output at 100 rows. The truth, because the columns are correlated, is 100,000 rows. Now the join-order search (§5) and the access-path chooser (§4) make their decisions on the *100-row* fantasy: an index-nested-loop join looks perfect, because doing 100 index seeks is cheap. The plan ships. At runtime it does not 100 seeks but 100,000 — a random-I/O bloodbath — and the query that should have taken a hundred milliseconds takes ten minutes. Nobody changed the data or the SQL. One bad estimate near the leaves chose a plan a thousand times too slow. This is the optimizer's characteristic failure: not a crash, not a wrong answer, but a **right answer produced catastrophically slowly**, and it traces back to a number the optimizer guessed before it read a row.

The causes are a catalog of ways the estimate drifts from reality. **Stale or missing statistics**: the table grew or its distribution shifted since the last `ANALYZE`, so the histograms describe a database that no longer exists. **Correlated columns**: the independence assumption, the single most common culprit, turning a product of selectivities into a number off by orders of magnitude. **Parameter sniffing**: the optimizer plans for one parameter value (`country = 'JP'`, common) and caches that plan, then reuses it for a wildly different value (`country = 'VA'`, vanishingly rare) where a totally different plan was wanted. And underneath all of it, the **long tail**: most queries get a fine plan, but the small fraction that get a disastrous one dominate the pain, and they're exactly the queries with the gnarliest estimation problems.

The honest assessment is Leis et al.'s, in their 2015 paper bluntly titled *"How Good Are Query Optimizers, Really?"*<Footnote n={2} /> Their finding, after measuring real systems hard: **cardinality estimation is the dominant source of error** — the cost model's details matter far less than getting the row counts right, and a good cost model fed bad cardinalities still picks bad plans. It is the empirical backbone of everything this chapter has claimed about §3 being the heart.

So what do real systems *do* about it? **Better statistics**: run `ANALYZE` regularly (or let auto-analyze do it), and gather **multi-column / extended statistics** that capture correlation directly instead of assuming independence away. **Hints**: when the optimizer is incorrigibly wrong, tools like `pg_hint_plan` let a human override its choices — an admission of defeat, but a useful one. And **adaptive execution**: re-plan or correct course at *runtime* when the actual row counts come in wildly different from the estimates, closing the loop the static optimizer couldn't.

<Callout type="note" title="In production: feed the optimizer, or it starves">
The single most common cause of a sudden, mysterious slow query in a healthy database is **stale statistics**. A nightly batch loads ten million rows, but `ANALYZE` hasn't run since the table had ten thousand — so the planner still thinks the table is tiny, picks a nested-loop that was perfect at 10K and ruinous at 10M, and the query that ran in 50ms yesterday runs for twenty minutes today. The fix is almost never to rewrite the query; it's to update the statistics (`ANALYZE table;`) so the optimizer is reasoning about the database that actually exists. The optimizer is a model of your data — keep the model current or it will confidently plan for a database you no longer have.
</Callout>

The sandbox takes the §5 join-order optimizer and feeds it a deliberately **wrong** cardinality — simulating the correlated-column underestimate above. Believing a correlated join will yield a single row when it really yields a hundred thousand, the optimizer schedules that join first and pays a true cost around 100,000, choosing the wrong order because it thinks an intermediate will be tiny when it's actually enormous. Then "fix the stats": replace the bad estimate with the true cardinality and rerun, and the *same optimizer, same code,* flips to the genuinely selective join first for a cost of about a dozen — a roughly nine-thousand-fold swing from one corrected number. Nothing about the algorithm changed — only the numbers it was fed. That's the whole lesson in one before-and-after: the optimizer is a model of reality, and when the model is wrong the plan is wrong, which is precisely why production databases spend so much effort keeping the model — the statistics — honest.

<Section
  id="whats-next"
  title="What's next"
>

Stand back and look at the engine now. It can find rows on a page (Chapter 04). It can keep the hot pages in memory and turn cold reads into cheap hits (Chapter 05). It can compile a declarative query into a tree of operators and run that tree, joining tables three different ways (Chapter 06). And now it can *choose* — survey the space of equivalent plans, score each one with a cost model fed by cardinality estimates, pick the index or the scan at the selectivity crossover, order the joins to keep intermediates small, and commit to a good plan in milliseconds before reading a single row. Everything you need to answer *one* query, and answer it well.

One query. That's the catch, and it's the one the rest of the book is about. A real database doesn't serve one query — it serves thousands, all at once, from clients who have never heard of each other, and a frightening number of them want to touch the *same rows at the same time.* Two transactions read the same balance, both add to it, both write back — and one of the updates silently vanishes. This is exactly the race that opened the whole book in Chapter 01, the lost update, the reason transactions exist at all. We've spent six chapters building an engine that can answer a single question beautifully. Now we have to make it survive a crowd.

> We can answer one query well. Now run a thousand at once, on the same rows, without them clobbering each other.

That's **Chapter 8 — locking and concurrency.** The engine knows *what* to do. Next it learns to do many things at once without letting them corrupt each other.
